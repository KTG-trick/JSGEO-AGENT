# CODEX.md

这份文档面向 Codex、Claude、其他 AI 编程助手，以及新接手项目的工程师。目标是帮助协作者快速理解 GEO-Agent Studio 的产品目标、流程边界、架构分层和修改约束。

## 项目定位

GEO-Agent Studio 是一个面向企业 GEO 优化的 Electron 桌面工作台。它帮助企业把自身资料整理成知识库，围绕 AI 推荐场景生成核心问题池，发现高权重信源，生成内容资产，完成半自动发稿，并在发布后持续检测 AI 推荐可见性，再把有效结果沉淀为下一轮优化规则。

项目的用户界面和主要业务文案使用中文。用户侧表达应保持清晰、具体、符合中国企业客户的工作习惯。

## 产品流程

当前主流程是 7 个阶段：

1. 阶段一：企业知识库构建  
   用户上传或录入企业资料，系统抽取企业关键信息，保留源文件资产，解析、清洗、切片，并建立全文检索与可选向量索引。确认后形成企业知识库。

2. 阶段二：AI 核心问题池  
   系统基于企业知识库、目标关键词、行业和区域生成 10 条核心推荐/排行榜类用户问题。阶段二是后续信源发现、内容生成和可见性检测的共同输入。

3. 阶段三：高权重信源发现  
   这是全项目唯一使用豆包助手联网能力的阶段。系统使用豆包助手联网搜索观察真实 AI 回答中可能参考的来源、平台和内容形态，形成信源优先级。

4. 阶段四：内容资产生成  
   系统基于知识库、问题池和信源方向生成支撑稿、测评稿、咨询稿和排行榜稿等内容资产。内容资产先进入稿件管理，不直接发布。

5. 阶段五：稿件管理与发布  
   用户校对稿件后，系统生成阿里云 OSS 公网预览链接。用户选择媒体资源并确认投递后，系统调用超级媒介发稿 API 创建订单、同步订单状态，并在发布成功后保存发布 URL。

6. 阶段六：AI 推荐可见性检测  
   有已发布 URL 后，系统自动检测核心问题下目标企业是否被推荐、引用或排名上升，并按间隔复查。这个阶段使用模型加 web search 插件，不使用豆包助手联网。

7. 阶段七：反思优化/自动学习  
   仅当阶段六命中已发布文章 URL，或同一问题下目标企业排名上升时，系统自动生成待确认学习规则。规则必须由用户确认后才进入下一轮内容生成。

## 架构速览

项目是 Electron 桌面应用，分为主进程和渲染进程：

- 渲染进程位于 `src/renderer/`，使用 React、TypeScript、Vite、Tailwind CSS 和 shadcn/Radix 风格组件，负责页面、对话、表单、稿件管理和状态展示。
- 主进程位于 `src/main/`，使用 CommonJS `.cjs` 文件，负责本地数据库、文件资产、文档解析、知识库索引、LLM 调用、OSS 上传、超级媒介 API、IPC handler 和后台服务编排。
- 两端通信统一通过 `window.geoAgent` 暴露的 IPC 能力，不直接跨层调用。
- 流式任务使用 requestId 和独立事件 channel，事件通常包含 meta、status、delta、reasoning_delta、result、done、error 等类型。

主进程服务按能力拆分。常见服务包括知识库、对话、项目、模型策略、LLM 网关、问题池、信源发现、稿件生成、发稿、OSS 预览、可见性检测和反思优化。

## 核心能力

### 知识库与 RAG

知识库不是只保存抽取后的企业字段。上传附件后，系统应保留原始文件资产，并将可解析内容转为文本、清洗、切片、建立全文索引，并在 embedding 可用时建立 sqlite-vec 向量索引。检索入口应优先保持稳定，让后续阶段自然获得更准确的资料片段。

当前知识库增强目标是轻量、够用、可追溯，不做复杂知识图谱和重型事实审核流。

### 对话与知识库关系

知识库是项目容器，对话必须归属到明确的知识库项目。创建第二个或更多知识库时，应先创建空知识库项目和唯一标识，再创建属于该知识库的新对话，后续抽取、确认和阶段提示都写入这个新对话，避免历史记录被拆分或混入旧知识库。

更新已有知识库时，应继续使用当前知识库下的对话，不应跨项目复用会话。

### 稿件发布闭环

稿件管理是半自动发布工作台。用户先校对文章，校对成功后生成 OSS 预览 URL。用户点击发稿时，系统才调用超级媒介 API 创建订单。AI 可以推荐适合的媒体渠道，但不能自动扣费下单，最终投递必须由用户确认。

发布成功后，发布 URL 会进入后续阶段六推荐可见性检测。

### 自动学习

自动学习承载阶段六和阶段七。阶段六自动检测和定时复查，阶段七只在命中文章或排名上升时生成待确认规则。自动学习结果不能直接污染基础企业事实，只能作为优化证据和下一轮内容策略依据。

## 模型与联网边界

这是项目里的高优先级约束：

- 只有阶段三“高权重信源发现”使用豆包助手联网能力。
- 阶段二问题池、阶段四内容生成、阶段六可见性检测、普通联网问答等，需要联网时使用模型加 web search 插件。
- 豆包助手联网成本更高、效果更强，只用于需要观察真实联网信源的场景。
- DeepSeek 路径默认不等同于 Ark web search 插件，除非后续明确接入可用搜索能力。
- 模型路由应集中通过任务策略控制，不要在业务服务里随手硬编码昂贵联网模式。

## 开发命令

- `npm run dev`：启动 Vite 开发服务和 Electron 桌面端。
- `npm run typecheck`：运行 TypeScript 类型检查。
- `npm test`：运行现有 Node 测试。
- `npm run build`：构建渲染端生产产物。
- `npm run dist`：构建并打包桌面应用。

只改文档时通常不需要跑构建。涉及主进程、渲染端类型、IPC 或业务服务时，至少运行类型检查；涉及服务逻辑时运行测试；涉及 UI 或打包风险时运行构建。

## AI 协作规则

- 默认使用中文交流，用户可见文案保持中文。
- 主进程只能使用 `.cjs` 和 CommonJS 风格；渲染端使用 `.ts`、`.tsx` 和 ESM。
- 不要随意重构相邻代码。优先做外科手术式修改，让每一处变更都能对应用户需求。
- 不要回滚用户已有改动；工作区可能是脏的，先理解再修改。
- 新增 IPC 能力时，需要同时考虑主进程 handler、preload 暴露、渲染端类型和调用方。
- 修改字段、流程阶段、模型策略时，要同步检查知识库详情页、智能助手、稿件管理、自动学习等使用方。
- 修改联网策略时，必须确认没有把非信源发现任务切到豆包助手联网。
- 涉及文件上传、OSS、API Key、企业资料时，不要把本地路径、密钥或内部 metadata 暴露到用户可访问页面。
- UI 改动应遵循现有设计系统，不做营销式落地页，不添加与工作流无关的装饰。
- 结束任务前说明验证结果；如果没跑测试或构建，要明确说出原因。

## 常见入口

- 智能助手：用户通过对话完成知识库录入、阶段提示和任务推进。
- 知识库详情：查看企业字段、健康评分、源文件、索引状态和 GEO 阶段进度。
- 稿件管理与发布：校对稿件、生成 OSS 预览、选择媒体、投递和同步订单。
- 自动学习：承接阶段六可见性检测和阶段七反思优化。
- 仪表盘/项目页：展示企业项目、整体状态和关键指标。

## 协作者心智模型

这个项目不是单纯聊天应用，也不是单纯 CMS。它的核心是“企业知识库驱动的 GEO 工作流”：

企业资料进入知识库，知识库生成问题池，问题池驱动信源发现，信源发现指导内容资产，内容资产进入半自动发布，发布结果再进入可见性检测和自动学习。任何改动都应优先保护这条主链路的清晰性、可追溯性和阶段边界。
